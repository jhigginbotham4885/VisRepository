3.1.1 Functional Objective 
Transmission Format is attached with Client profile .It can be attached for state, client and carve out. Data is written in outbound member according to the transmission format attached.

Field Level: (EFD Screens) - Field level setup for all the fields 
•	CONSTANT VALUE Setup : Optional 
At field level setup constant value can be set on EFD screen. When constant value is provided for any field then same value is populated in O/B member for the field. 
 
•	MAPPING Setup: Optional 
If constant value is not given to a field then mapping is selected to the field. Data is populated in that field according to logic written for attached mapping. 
 
•	TRANSLATION Setup: Optional  
When mapping is selected for a field then value populated through the mapping can be translated to any given value using translation setup.

3.2 Segment Information  
3.2.1 Functional Objective 
A segment may be defined as a collection of fields of similar nature. Segments are used to group information into logical chunks. Each format may contain one or more segments. Every segment in the transmission format has a unique ID for identifying it. While creating a transmission format from a base format, user can select or unselect the segments according to requirement. 
Segments can be categorized into two broad categories namely: 
Looping: These are the segments that are populated more than once in the outbound with claim information in loop. 
Non-looping: These are the segments that are populated once in the outbound. 
 
Segments can also be categorized according to the information they store. These are: 
Header segments: Segments that contain header information like creation date and time fall in this category and are present at the beginning of each record in outbound. 
Claim Detail segments: Segments that contain claim information like pharmacy number, claim number fall in this category. 
Trailer Segments: Segments that contain trailer information like record count and batch number fall in this category and are present at the end of each record in outbound. 
Segments are present in both fixed and variable length transmission formats. In Fixed length formats, the segments have fixed start and end positions irrespective of the length of data. While in variable length formats, the position of segments is not fixed.

3.3 Fields 
3.3.1 Functional Objective 
Fields are independent entities for holding the actual information. Each segment may contain one  or more fields that form a logical group. Like segments or formats, fields cannot be further subdivided. Every field in EMS has a Field ID that is used to uniquely identify the fields are categorized in the following: 
•	Header fields: These fields are present in the header segments and are populated at the beginning of the record in outbound. Loop flag for header fields is ‘H’. Like Record type field in the header segment of MEDS III format 
•	Trailer fields: These fields are present in the trailer segments and are populated at the end of the record in outbound. Loop flag for trailer fields is ‘T’. Like record count field in the trailer segment of MEDS III format. 
 
According to the number of occurrences of the field in outbound, the fields are categorized in the following: 
•	Looping fields: The fields that are populated more than once in a segment are called looping fields. Looping fields can be identified by the loop flag as ‘I’, ‘J’, ‘K’ or ‘L’. 
 
•	Non-Looping fields: These fields are populated only once in the segment. Non- Looping fields are identified with the loop flag as ‘N’. 
 
Fields can be categorized on the basis of the type of information they store (Data Type) as:  
•	Alphanumeric fields: These fields can store both character and numeric data including blanks. 
•	Numeric fields: These fields are used to store numeric data. 
•	Dollar fields: These fields are used to store price information (Both positive and Negative). They can contain numbers and special characters.

3.4 Mappings  
3.4.1 Functional Objective 
Mappings are used to extract the format field values form the EMS system. Claim level information  is present in the EMS system in various files and mappings are attached on the fields of a transmission format to extract the appropriate values from the EMS files into the respective fields. 
Every field has an associated mapping that is used to retrieve data for the field and populate it in the outbound. New mappings can be created as per the requirement. A mapping must be available on the field for it to be attached. A field can have multiple mappings available for it, but at a time only one mapping can be attached on the field. Same mapping can also be available for two fields. A constant can also be set on the field, in which case, there will be no mapping attached on the field. 
 
3.4.2 Examples/Scenarios 
•	Mapping PLNCDE2 on field 270B is used to extract the account ID from ECA file and populate in outbound. 
•	Mapping DTOBRT3 on field MDCD12 is used to extract the data from the date of birth field of ECA file and populate in outbound.

4 Daily Extract to EMS extract Process 
4.1.1 Functional Objective 
 
 
 
There is a nightly extract which pulls the data from RxClaim and sends to EXP1, once this job is complete DMEDEFE extract job runs to load the data into EMS. Execution of DMEDEFE is effected by the volume of the data. Due to high volume of data being processed from RxClaim extract to EMS and its growing needs, at times DMEDEFE runs longer which in turn delays the kick-off of outbound encounter generation jobs. To address the performance, the driver program (RCECA001) used by DMEDEFE is enhanced to support multiple threads that will process a different set of independent claims. 
  
Job will be processed in 2 portions. 
o	Portion-1: will pick the C/A/G records from C/A/G List file in sorted order (Record Count – Higher to Lower) and write to DataQ. 
o	Portion-2: multiple instances of job will be submitted to achieve faster execution as shown in the above figure. C/A/G from DataQ will be picked and passed to RCECA001 program 
o	RCECA001 program will search the C/A/G in EX1P file and process as per existing process. In addition to this RCECA001 program will populate the record count at C/A/G level in C/A/G List file. 
o	Separate job queues shall be available for the system to run multiple jobs at the same time. 
o	Driver Program shall not be able to change the number of child jobs dynamically once parent job is submitted. 
o	Maximum job count for child jobs could be 99. Currently, it has been set to 5.

4.1.4 Technical Details 
o	Driver Program (RCDME001) 
1.	Call program RCDME002 to retrieve Unique CAG list and populate the same in DataQ in the sorted order of claim count.

5 EMS Claim Processing 
 
5.1 Outbound Processing 
5.1.1 Functional Objective    
Outbound Processing is the core objective of the EMS System. When we say outbound processing, it means that EMS System will process the adjudicated claims and will send out successfully processed claims in a file at a particular location (usually setup by Business). 
 
The processed claims in the file will be analyzed by the business. If the claim data in file is as per expectation the claim will be approved. For any error in claim data, the claim will be rejected and will come back to EMS System for next outbound generation after correcting the errors (The process of rejecting/approving the claims is yet another objective of EMS System, i.e. Inbound Processing) 
 
File Type: The file in which the claims are sent is usually a Flat File of 8000 bytes length per record to hold the claim/record data  
 
File Formats: When processed claims are sent out in a file they are sent in a particular format or layout. In EMS system, the claims are sent in two different formats namely, 
•	Fixed Format 
•	Variable Format 
 
Fixed Formats File: As the name suggests, fixed format claim data lies at a fixed position in the file. Each claim details may be stored up to maximum in 3 segments or 3 rows. The three segments where the claim data is stored are: 
	 	 	DE- Detail Segment 
	 	 	CD- Compound Segment 
	 	 	CE- Extended Compound Segment 
 
Variable Formats File: In variable format file, the claim data is not stored at a fixed position. The position of the data will depend upon the processing of fields. (Suppression, Truncation etc.). Each claim detail is stored in one row only (all segments are populated in a row). 
 
Data Modifications: Whenever we sent a claim’s data in the file, it is usually processed before writing in the file. The various formatting options which are applied on the claim data are given below  
•	Extraction 
•	Transformation 
•	Formatting 
•	Bad/Held Processing 
 
5.1.1.1 Extraction 
When we extract the claim data from the different Adjudication files into the EMS files, the process is called extraction 
 
 	 
 
5.1.1.2 Transformation 
As the name suggests, the data is transformed before writing in the flat file. The different transformation which is applied on the data are: o Default Values Translation o Lower to Upper case transformation 
 
5.1.1.3 Formatting 
The different formatting which is applied on the data are: 
o Applying Over punch to transform a decimal/numeric data o Zero Padding on numeric fields o Suppressing a Blanks Field 
 
5.1.1.4 Bad/Held Processing 
A claim never goes in the outbound file unless it satisfies the Bad/Held rules or conditions. The Bad/Held rules are applied to do away with those claim which are not eligible to be transmitted in the outbound.  
Bad Rules: Once a claim is marked as Bad, it can never be transmitted under any circumstances in the future. For example, if a ‘claim does not have a Member Id’, it is meaningless to process it in the next outbound run also. There are is list of Bad rules setup by business which can be applied on the claim 
 
Held Rules: A held claim can only transmit if it satisfies the held conditions in the next outbound cycle. For example, if a ‘claim has not crossed the minimum aging period’, it will be marked as held. This claim can only transmit after satisfying the minimum aging period 
 
	5.1.1.5 	Pre-Edit Processing 
Pre-Edit Validation is a process where EMS System validates/checks fields of a claim so that the claims which are failed in pre-edits will not be placed in the encounter/outbound file immediately. Instead these claims are marked with different status in EMS for reconciliation. These pre-edit or pre-scrubbed claims are treated as EMS Rejected claim and reported in an exception report.  
 
	5.1.1.6 	Outbound Generation 
When all the above rules or conditions are applied on the claim, it is ready for transmitting in the outbound run. The process is called as Outbound Generation. 
Once Outbound is generated, it populates the various other EMS files like ECT, CRP, and EBH to keep track of the claims sent.

5.2 Pre-Edit Functionality 
5.2.1 Functional Objective 
Pre-Edit Validation is a process where EMS System validates/checks fields of a claim so that the claims which are failed in pre-edits will not be placed in the encounter/outbound file immediately. Instead these claims are marked with different status in EMS for reconciliation. These pre-edit or pre-scrubbed claims are treated as EMS Rejected claim and reported in an exception report. 
 
This is a setup driven process where client can set up Pre-Edit/Pre-Scrubbing rules Flag on Client Profile to setup the Segment and Field Value Pre-Edit Rule for MEDS III, NCPDP D.0, Post Adj Four Formats, MED STAR, XML 01.00 formats. Default value N will appear for each segment & field level rule.                                
 
Highlights of Pre-Edit Rule 
Segment Level (ESP Screen) 
•	B1, B2, B3, B4 Required Flag 
 
Field Level: (ENP Screens) 
•	Lmt Val Flag Y/N – OPTIONAL 
•	Rqd Flag Y/N – OPTIONAL 
•	Lft Jst Flag Y/N – OPTIONAL 
•	Rgt Jst Flag Y/N – OPTIONAL 
•	Num Flag Y/N – OPTIONAL 
•	Ld ZrFlag Y/N - OPTIONAL 
 
 
o	LIMITED VALUE Check : Lmt Val Flag Y/N – OPTIONAL 
If value is ‘Y’: The corresponding field should have the value as specified under the column on ‘Limited Value’ screen. If multiple values present under constant value then the corresponding field should have any one of the value in an encounter file.  
 
Limited values for a field can be defined by taking option 2 on that field. For a field, more than one limited value can be defined. 
If value is ‘N’: No Limited Value check will be applied. 
 
o	REQUIRED VALUE Check : Rqd Flag Y/N - OPTIONAL 
If value is ‘Y’: The corresponding field should have some value in an encounter file. For Numeric fields, 0 is also considered as a value. 
If value is ‘N’: No Required check will be applied. If field is blank, no other edit-rules will be applied but if field is non-blank, other edit rules will be applied. 
 
o	LEFT JUSTIFIED VALUE Check: Lft Jst Flag Y/N - OPTIONAL 
If value is ‘Y’: The field should be placed a value with left justification in an encounter file. 
If value is ‘N’: No Left justification check is required. 
 
o	RIGHT JUSTIFIED VALUE Check: Rgt Jst Flag Y/N - OPTIONAL 
If value is ‘Y’: The field should be placed a value with Right justification in an encounter file. 
If value is ‘N’: No Right justification check is required. 
 
o	NUMERIC VALUE Check: Num Flag Y/N - OPTIONAL 
If value is ‘Y’: Indicates the field value type should be numeric type of data. And the value sent in an encounter file should be a Numeric value. 
If value is ‘N’: Indicates No pre edit check will be applied for the corresponding field and field value can be either numeric or alphanumeric. 
 
 
o	LEADING ZERO VALUE Check: Ld ZrFlag Y/N - OPTIONAL 
If value is ‘Y’: It should be a Numeric value and leading zeros should be filled if the actual field value digits are less than the specified field length. 
If value is ‘N’:

5.4 Reconciliation 
5.4.1 Functional Objective 
This process has been designed in EMS system to correct the Error claim details sent to State in attached outbound format. There are two scenarios under which reconciliation is required:  
1.	When Claim is rejected by Inbound Process  
2.	When Claim is rejected by Pre-Edit Process 
 
5.4.2 Examples/Scenarios 
In Outbound Process, Claim Details are retrieving from EMS File and Rx-Claim File. It populates the details under the different Field Id of Format/Vr, either the information sent in claim is having wrong information or *Blank Information. The defined process at State will reject the Claim and send the reason through Reject Code. The Error Code will help the business to do correction through Reconciliation Process. Similarly if the outbound string populate is having some invalid or *blank data. It will Pre-Scrubbed the Claim and give the respective Pre-Edit Error Code that will get corrected through Reconciliation Process.        
 
5.4.3 Process Flow Diagram 
  
5.4.4 Technical Details 
The current reconciliation will be done through the different menu option of EMS Menu. It will be done through RCECT003, RCECT039, RCECT046, RCECR008-RCECR011, RCECR015 and RCECR016. 
The reconciliation process on screen will be done through option ‘2’ and only when correction needed flag is set to ‘Y’. Business user can remove the Correction Needed Flag to *Blanks, if partial correction has been done on the claim.  
Reconciliation Process Correct the Alpha-Numeric Field and Numeric Fields Only. It won’t allow business user to make changes in the Dollar Field of Outbound Format.        
Reconciliation Program Flow Details: 
The Reconciliation Process for Rejected Claim: Parse the Outbound String & populate the Field Level Details in FMT File. The reconciliation is being done through FMT Screen.  
-	RCFMTP Populate: Fixed Format RCECA038, Variable Format RCECA041 and MedStar Format RCECA047. 
Once the correction is being done by business user, the FMT File Field Level details will get De-Parse and populate the outbound string. The updated outbound string with corrected detail in Field Id will get populated in the State File.    
-	Program to Populate the String: Fixed Format RCECA039, Variable Format RCECA049 and MedStar Format RCECA048. 
 
Note: The Pre-Edited Claim that has never reported to State and the Format or Version get modified before the next outbound run or reconciliation. It would convert the string in the new Format/Version associated to Client Profile.

	7.2 Translation 	 
7.2.1 Functional Objective 
Translation functionality is used to provide field level translation on the transmission format. This functionality enables the business user to define and configure default or translate/crosswalk values at the field level.  
It is invoked after the field value is evaluated from the mapping. It is available for both fixed and variable length transmission format fields. The translation functionality is invoked from the EFD screen with option ‘T’ to maintain translation. 
When a user specifies option ‘T’ against a field on the EFD screen, the user can specify the translation ‘From’  and ‘To’ values. User can either specify a constant on the field or provide translation on it. Both constant and translation setup cannot exist simultaneously for a field. Translation setup can be done on all types of fields, that is, Alphanumeric, Numeric and Dollar fields. The translated value of a field cannot be greater than the field length. 
 
7.2.2 Examples/Scenarios 
Consider a field 436-E1 of type Alphanumeric. The translation setup on the field can be as per below table: 
From Value 	To Value 
A11 	A12 
B11 	B22 
C23 	D14 
E67 	abc 
 
As  given above, a field can have multiple translations set on it and the From and To values are case sensitive, that is, if the ‘From’ value on a field translation setup is ‘abc’ and the value evaluated after mapping is ‘ABC’, the field value will not be translated. 
 	 	 	 	 
From Value 	To Value 
35 	45 
35 	47 
 
For each field, there can be only unique set of from values in the translation setup. For example the  setup given above is invalid and cannot exist for a field as the same From value cannot be translated to multiple ‘To’ Values. 
 
Also, the translation table can have blank values for ‘From’ and ‘To’ values provided the field type is alphanumeric. 
•	For alphanumeric fields, the translation setup can have blank values, special characters and numbers. 
•	For Numeric fields, the translation setup can have only numeric values. 
•	For Dollar fields, the translation setup can have special characters and numbers but blanks are not allowed.

7.3 Void Process – Void/NFIX 
7.3.1 Functional Objective 
The need for void functionality arise as certain encounters were not being submitted to the Medicaid state agency because they did not meet encounter requirements and these encounters were not identifiable and prevent business user’s ability to accurately report encounter status to the clients. Therefore, an internal mechanism was needed whereby accepted and rejected claims can be tracked and voided. 
 
Void functionality provided the ability for the business to void a rejected claim as well as an accepted claim by introducing a new reject code in RxClaim EMS system.  When a claim is flagged with this new reject code, no further iteration of this claim will be released in the outbound file. Functionality was built to allow a user to change the new reject code to a different value so that a user can release subsequent iterations of these voided claims if needed. These voided claims would be sent to the outbound file with correct record status codes/transaction codes which will vary according to the format used by the client.  
 
Highlights of Void/Non-Fix Functionality: 
‘V’ (Void)  o The claim should be either state Approved/Rejected o The claim should not be a normal B2 or Self-created B2 record o To void a claim, it should be a recent transaction. 
E.g. if B1 (999) and B1 (998) exist, Void cannot be taken on B1 (999) o Once a claim is successfully voided:  
•	Display Status and Correction needed flag is updated to ‘NV’ and ‘Y’ respectively of the voided claim in ECT file. The display status and correction needed will be visible on following screens RCECT003, RCECR008, RECR009, RCECR010, and RCECR011. 
•	Record status of voided claims will be updated to ‘R’ in ECT file. For claims having reject code as ‘HR’, Record status will remain same.  
•	‘VOID’ Error Code entry will be created in ECR file. For all voided claims which have error code as ‘VOID’, will never be transmitted in outbound  
•	All the claim iterations will be put on hold. The functionality is achieved by creating an entry of the claim in Claim Exclusion File (ECZ) with ‘000’ sequence. This will hold all the subsequent iterations of the claim from going in outbound. 
•	Status of record is updated to ‘H’ for subsequent fresh iteration of void claims & ‘V’ for the voided claim number/Sequence in ECA File. 
•	Claims counts in EBH file will be updated, if an Approved claim is voided. The approved claim will now be treated as rejected claim after void and hence Rej. Claim count be increased and Approved claim count be reduced 
o Self-Created B2 should be transmitted to state for the voided claim in next outbound. 
 
 	 
 
‘N’ (Non Fixable Reject)  o The claim should only be state Rejected. For Approved claims Non-Fix is not allowed o Non-Fix can be taken on normal B2 and Self-Created B2 records. o Once a claim is successfully Non-Fixed:  
•	Update display status to ‘NV’ in ECT file. The display status will be visible on following screens RCECT003, RCECR008, RECR009, RCECR010 and RCECR011. 
•	User can't update correction needed flag to 'Y' for Non-Fix Claim.  
•	Record status of Non-Fixable claims will remain ‘R’ in ECT file. For claims having reject code as ‘HR’, Record status will remain same.  
•	'NFIX' Error Code entry will be created in ECR file. For all Non-Fixable claims which have error code as ‘NFIX’, can be transmitted in outbound only after fixing the claim.  
•	All the claim iterations will be put on hold. The functionality is achieved by creating an entry of the claim in Claim Exclusion File (ECZ) with ‘000’ sequence. This will hold all the subsequent iterations of the claim from going in outbound. 
•	Status of record is updated to ’H’ in ECA file for Non-Fixable claims o Self-Created B2 record is not created for Non-Fixable claims. 
 
‘R’ (Release) o Release option is only allowed on claim having ‘VOID’ Error Code. o Once a claim is successfully Released 
•	Display status of the claim is updated to ‘VR’ in ECT file. This Claim + Claim Seq.  Never transmitted in outbound (either B1 or B2) & User can't update correction needed flag to 'Y'.  
•	Claim next iterations will get transmitted in next outbound.  
Update Claim Sequence Number from 000 to specific sequence number in claim exclusion file. This will ensure that the voided Claim + Claim Seq will not get transmitted and all claim next iterations will get transmitted in outbound. 
•	Status of record will remain ‘V’ in ECA file for the voided claim since it will never transmit in outbound. 
•	Error code entry in ECR file for the claim will not be removed. 
‘F’ (Fix) o Fix option is only allowed on claim having ‘NFIX’ Error Code. 
o Once a claim is successfully Fixed 
•	Display status of the claim is updated to prior status in ECT file. 
•	Claim + Claim Seq. will be ready to get transmitted in outbound, once user update the correction needed flag to 'Y'.  
•	All the claim iterations will be available to get transmitted in next outbound. This functionality is achieved by removing the claim entry from ECZ file 
•	Claim Status of record will be changed to ‘R’ in ECA file.   
•	Error Code 'NFIX' should be removed from ECR File.

7.6 File Regeneration 
7.6.1 Functional Objective 
Prior to File Regeneration functionality, business was not able to submit correction file in latest transmission format version, when required to be compliant with State requirement. 
With File Regeneration functionality Business now have the flexibility to generate correction files either in previously submitted encounter transmission format versions or in the latest transmission format version. 
Previous Transmission Format Version: The format version which is used in most recent encounters sent to State.  
Latest Transmission Format Version: The most up-to-date format version available to use to send encounters to State. 
 
 
7.6.2 Examples/Scenarios 
Few claims were transmitted for a client profile “Client1” having Outbound Format id “Format1”. After transmission user came to know that he has transmitted the claims on wrong Outbound Format id or specifications of outbound format id have been changed. In this case user changed the outbound format id on client format “Format2”. Now format id become different in CRP file and ECD file for same client id.  
By using File Regeneration functionality user can change the outbound format id of transmitted claim with the format id attached on Client id that is Format1 to Format2.

With group maintenance – 
1.	The Reconciliation team will have the ability to group claims for correction in submission groups.  
 
2.	Groups are maintained in a table RCEGRP consisting of Group Name, User ID of creator, Description of Group, Released for transmission flag (F9 and Next Cycle). 
 
3.	A group maintenance program is available for RCEGRP table which allows to:   
o	Add/Delete group’s 
o	The resubmission grouping has the capability to hold/release the corrected encounters. A released group can either be immediately placed on an encounter file or made available for the next outbound file. 
o	Count of total claims in each group is visible. o Release for transmission flag 
i.	Standard cycle only (F9 does not affect) 
ii.	Flag can be modified to Y if claims are ready to be transmitted.

1.	Understanding of the project 

New features which are added for looping segments are as below
•	Parent Child Relation

1.1	Parent-Child Relationship 
 New looping segment fields have been arranged to work in a parent-child relationship model which will be used during the encounter generation process. 

o	The system internally handles this relationship and cannot be changed from any user screen.
o	Any parent field will be seen as highlighted in Yellow color. User can further take option ‘7’ on the parent field to see and setup the corresponding child fields. 
o	If Child field has further child fields, user can further take option ‘7’ on that field to see and setup the sub-child fields. 
o	If Parent field is unselected, the child fields (even if selected) will not be populated in outbound file.

For example, consider field’s 337-4C, 341-HB, 342-HC, 431-DV, 353-NR, 351-NP and 352-NQ of AM05 segment. 

337-4C -> Parent Field

341-HB -> Child Field of 337-4C and Parent for below fields
342-HC -> Child Field of 341-HB
431-DV -> Child Field of 341-HB

353-NR -> Child Field of 337-4C and Parent for below fields
351-NP -> Child Field of 353-NR

1.5	Loop Accumulation

o	Current system does not have the ability to accumulate loops in the encounter file since it send all the loops.
o	Loop Accumulation functionality accumulates only Dollar amount fields present in the set of looping fields. Example in case of HB, HC and DV: Only values for DV field will be accumulated because DV is defined as D (Dollar) type field on transmission format.

1.8	 General understanding 
1.8.1	Switching from old looping system to new looping system
o	New looping segments will not be visible on existing transmission format in EMS.
o	New looping segment will be available only when user will create a brand new transmission format or if user will copy an existing transmission format to a new version.
o	All the new looping segments will always be unselected by default.
o	All the new looping segments and fields will be having default mappings and default setups and user will have to setup the transmission format for looping fields as per the need.
o	Once a transmission format with new looping segments is attached on a client profile, user will have to take option F17=PrEdt Setup on client profile setup at least once to refresh the pre edit setup to include the new looping segments.
o	Once the pre edit setup is refreshed, user will be able to set up pre edit like regular fields for the looping fields. All the new looping segments and looping fields will be visible on pre edit setup like regular fields.
o	All the pre edit flags will be set to default N for all the new looping segments and fields.
o	While setting up Pre edit for a new looping segment, user will have to remove the existing pre edit setup from the old segment. This is due to existing pre edit functionality that is pre edit rules will be applied even if a segment is not selected on the transmission format.
o	Existing mappings cannot be used interchangeably for new looping segments/fields. We have created new mappings for new looping fields and list is attached in this document. Also, there is a document which will show old and corresponding new mapping for all looping fields.
o	New mappings created for a particular looping field cannot be used on other looping field in the new setup.
o	If parent counter field is defined as constant and no mapping is attached, child fields will not be populated from the RxClaim source and child fields should also be set as constant.
o	For format AETNA LA SUPP, there is no change in the looping project because this format works from the encounter of state format on the profile. Basically AETNA LA SUPP format is out of scope of this project.

12.	Response File Standardization functionality

Message based architecture is used to process claim responses using MQ series interface, All the Inbound response from various state/Client is consolidated in Web system and send a single format to EMS system. Inbound response will be delivered in MQ series queue from web system with all necessary information to update claim status in EMS files.


Process flow 

 

1)	New Files: Below new files have been created for the process

•	WKIN1P - This file stores the details corresponding to the client profiles processed in the Response file along with detail of Group exception, inbound exception and New Error code report members tagged to each Client Profile.

•	WKRSPP - This file stores the details of all the records received from the response queue. The records in the file will then be used to send acknowledgements through the acknowledge queue. The Processed flag of the records will change from Blanks to ‘R’ upon completion of this handshake process.

Additionally, the records of this file will get purged using the utility WKRSP001 which will use either the SF1 rule or input parameters to purge the records on the basis of their change date.

•	WKEBHP - This file stores the details of Approved/Rejected claim count corresponding to each Client Profile and Batch number along with the trailer record details. The records in this file will get deleted after processing the EBH/BEL/Metric file for the response file.

•	RCBE1P - Record will get populated in this file only when inbound processing happens with the new MQ Process. 

•	TDXEP1 - New File at location APIMEDITRN for storing the inbound exception report members.

•	TDXEP2 - New File at location APIMEDITRN for storing the New Error Core report members.

19.	Archival Restore process 

Below is utility command which is used to restore the archival data of EMS file 


SBMJOB(CALL PGM(RCEMS001R) PARM('Request_ID' 'Input_Library' 'Files_Flag' 'Report_Flag' 'AL4_Flag' 'Purge_Flag') )

Parameters

1)	'Request_ID' : Can be IM# of request from business. (Max 12 bytes)
2)	'Input_Library' : Library for all input file.
3)	Files Flag 'NNNNNNNNN'
Position 1 : ECA to EA1-EA7 (input file : EMSECAP)
Position 2 : EA1-EA7 to EA1-EA7 (input file : EMSEA1P – EMSEA7P)
Position 3 : ELF to ELF (input file : EMSELFP)
Position 4 : ECR to ECR (input file : EMSECRP)
Position 5 : HEL to HEL (input file : EMSHELP)
Position 6 : ECT/EHT to ECT/EHT (input file : EMSECTP/EMSEHTP)
Position 7 : AL2 to AL2 (input file : EMSAL2P)
Position 8 : CRP to OB1-OB9/OX1-OX2 (input file : EMSCRPP)
Position 9 : OB1-OB9/OX1-OX2 to OB1-OB9/OX1-OX2 (input file : EMSOB1P-EMSOB8P / EMSOX1P-EMSOX2P)
4)	'Report_Flag' : Creates business report at APIMEDITRN/EMS_REPORT(EMU*) from report file “RCEMUP”. Production Support to validate the file data RCEMUP.
5)	‘Al4_Flag’ : Uses input file “EMSAL4P” to update RCAL4P file with status as “U” (Unarchived).
6)	‘Purge_Flag’ : Used to purge data in RCEMUP table. Data older than X number of days. X days are soft coded in SFM rule “PRGEMU”

21.	EMS New Runtime Functionality 

New functionality introduced to handle the Runtime logic from following screens:
RCEFD001
RCEFD003
RCEFD026
RCEFD027
RCEFD027
RCEFD029
RCEFD030
RCEFD031
So that, business can control the Runtime functionally with Runtime Capable mappings without that needing to be hardcoded.

Runtime Field – Editable (Y/N)
The Runtime setting controls whether runtime logic is executed at Format Field levels.

If Runtime = Y on a Field and you change the Mapping to one that does not support runtime (Runtime = N),
? The user must first change the Field Runtime to N.

If Runtime = N on both the Mapping and Field,
? The Runtime Field becomes non-editable.

Runtime Logic Behavior
Y (Yes) – The runtime mapping reads RxClaim to update encounter data.
N (No) – The mapping does not read RxClaim but may read from the EMS Database.


Looping Fields
Looping Fields do not execute runtime logic.
For these fields: Runtime is automatically set to N, the setting is not editable
